# Risk register

| Risk | Current evidence | Mitigation / next proof |
|---|---|---|
| Speech accuracy and latency | Direct Apple transcription now runs through command and dictation paths, but no controlled user recordings have been measured. Upstream benchmarks are not representative. | Dogfood 0.4.0, run the fixed command and prose corpora, record partial/final latency and accuracy, then compare independent engines before closing decision 0003. |
| Permission reliability | The 0.4.0 target contains `NSMicrophoneUsageDescription`, only the audio-input entitlement, explicit microphone and Accessibility prompting, denial UI, and settings routes. Build 11 dogfooding confirmed that an old Accessibility row can remain visibly enabled while macOS rejects a newly ad-hoc-signed binary whose code requirement changed. Build 12 detects requirement drift, documents stale-row removal, and offers an explicit Topher-only reset flag; no reset occurs automatically. Build 18 was installed after its explicit Topher-only reset and re-approval. | Verify grant, denial, Settings recovery, relaunch, rebuild, and `/Applications` behavior again for the next installed artifact. Add a stable Apple Development signing identity before depending on permission persistence or distributing. |
| Speech asset availability | `en_US` support and installed/downloadable inventory are checked at runtime; installation progress and failure are visible. Download interruption, low disk, and concurrent inventory ownership are not yet exercised interactively. | Test installed, download-required, concurrent download, offline-after-install, interrupted download, and insufficient-storage paths. Keep manual text usable when voice is unavailable. |
| Audio route and analyzer recovery | Unit tests cover conversion, cancellation, stale results, stream errors, mode-specific maximum-duration finalization, finalization timeout, recoverable partials, and key-up races. Physical device changes, sleep/wake, and long dogfood sessions are unmeasured. | Exercise built-in, Bluetooth, and USB microphones; route changes mid-hold; sleep/wake; maximum-duration assistant/dictation holds; and 100 repeated sessions without restarting Topher. |
| System model availability | Runtime reports `appleIntelligenceNotEnabled`. | Deterministic commands are complete without it; gate every model use at runtime. |
| Application-name resolution | Build 8 discovers bounded conventional app directories and resolves exact normalized aliases to typed bundle identities. Colliding aliases fail closed, explicit app/site qualifiers control precedence, symlinked bundles are ignored, and execution re-resolves through `NSWorkspace`. Speech can still vary uncommon names, and catalog changes require relaunch. | Bias local recognition toward bounded installed display names; use personal vocabulary for known pronunciations; measure false opens; preserve explicit qualifiers and ambiguity failures; consider refresh only if real install-during-session usage warrants it. |
| Prompt injection | Future web, screen, and document text is untrusted. | Separate instruction/data channels; typed proposals; policy independent of model; never execute code/scripts from content. |
| Browser integration complexity | Build 22 keeps the narrow MV3/native-messaging adapter at regular-tab metadata/exact activation plus one YouTube Home schema. Required `scripting` plus optional `https://www.youtube.com/*` remain unchanged; only the extension popup can request/remove host access. Protocol v3 adds content-free live readiness, separate presentation/title completeness, bounded extension-internal uniqueness candidates, and target-only open revalidation. The primary app creates its authenticated local socket eagerly, but tab/page reads remain on demand. Automated tests cover identity/scope, secure setup, permission state, sanitized current/legacy DOM, hostile bounds, route/version mismatch, suspension/disconnect classification, cancellation, staleness, duplicates, target versus unrelated DOM drift, and one-shot/unknown mutation outcomes. The installed app and full Chrome command path remain unverified. | Run the Build 22 manual checklist on the exact built app. Verify setup/repair/conflict messaging, truthful disconnected/permission state, extension load/reload, denial, revocation, popup recovery, MV3 suspension, lazy loading, DOM changes, active-tab races, and a real feed read/open. Keep incognito unavailable and acquisition demand-driven. Require another ADR before any origin, route, extractor schema, browser effect, distribution identity, or persistence expansion. |
| Native-host registration and local impersonation | The checked user manifest binds one exact extension origin to an absolute helper in the current app bundle. The app validates the manifest and same-user launch-token handshake. Processes running as the same macOS account can still inspect that account's files and are outside a strong isolation boundary. | Keep the registration user-owned and non-group-writable, helper non-symlinked and executable, socket directory `0700`, token `0600`, and protocol typed/bounded. Revisit code signing, App Sandbox/XPC isolation, notarization, and distribution before sharing builds. |
| Sandbox posture | Hardened Runtime is enabled, but App Sandbox is disabled. Current web commands delegate navigation to a browser; the Chrome adapter adds a same-user Unix socket, native helper, and optional browser-owned YouTube page access but no direct network client or macOS privacy entitlement. | Revisit App Sandbox, helper/XPC design, capability entitlements, stable Developer ID signing, and denial/recovery behavior before direct networking, broader local data access, any additional DOM/page capture, or distribution. |
| Explicit domain navigation | A spoken domain can be misrecognized or intentionally point at an unsafe site even though Topher never fetches it directly. | Require an explicit navigation verb; force HTTPS; accept only bounded public DNS-style hosts; reject paths, credentials, ports, IPs, custom schemes, and local/reserved names; let known destinations win first; preserve action-correctness dogfood feedback. |
| Search query disclosure | Topher does not send search text to Unified Logging. During local dogfooding, transcript diagnostics default on, so the exact command—and therefore its search query—is retained locally. Executing the search also sends the query to Google or YouTube through the browser. | Keep providers and endpoints fixed, label the capability sensitive, execute only explicit typed searches, make the external handoff visible, and keep transcript diagnostics bounded with a persistent off switch and immediate clear. |
| Developer transcript retention | The dogfood-default-on developer trace deliberately stores exact finalized voice/manual commands and non-secure dictation, which may contain queries, URLs, pasted content, credentials, or other private text. Local storage is not equivalent to encryption or isolation from the same macOS account. | Show an orange enabled indicator; preserve explicit opt-out and require confirmation before re-enabling; exclude dictation targeting a secure or newly secure field; keep Unified Logging metadata-only; do not separately append audio, partials, retrieved context, constructed URLs, Keychain/config values, or detailed errors; enforce 24-hour, 200-record, 1-MiB, and 4-KiB bounds with restrictive POSIX modes and backup exclusion; invalidate previously issued tokens and prevent queued late records after disable/clear; provide immediate clear; never attach an unreviewed trace to a public report; review the default again before external distribution. |
| Durable local dogfood corpus | The explicit observed-query exporter creates a second plaintext copy that outlives the rolling 24-hour trace. Real commands may contain private text, and `--include-dictation` raises that sensitivity. | Never export automatically; keep the destination under gitignored `.topher-local`; exclude dictation by default; enforce owner, symlink, entry, phrase, and file bounds; deduplicate imported trace IDs; document intentional deletion; promote only sanitized cases to the public corpus. |
| Always-on energy | Push-to-talk opens the microphone only for an explicit hold; no ambient mode exists. Capture/analyzer resource use is not yet measured. | Measure warm/cold CPU, memory, energy, and asset retention during the corpus. Keep wake word and ambient capture out of MVP. |
| Personalized speech overreach | Alternative hypotheses or user vocabulary could turn uncertain speech into the wrong action. | Preserve raw text; accept only one uniquely supported alternative or an explicit vocabulary replacement that resolves to an allowlisted typed command; reject ambiguity; never let correction create URLs, bundle IDs, capabilities, or policy authority; benchmark false corrections and unintended executions at zero. |
| Ambient privacy and false activation | A future wake phrase would keep a local detector and its bounded rolling in-memory audio active and could activate accidentally. No ambient mode exists today. | Treat wake detection separately from transcription; continuously discard the detector window and never persist, log, transmit, or transcribe it before a confirmed wake; measure false accepts/rejects and idle energy; require an obvious enabled indicator and kill switch. |
| Remote chat authentication and replay | Discord/Slack/WhatsApp ingress would introduce provider credentials, remote identity, retries, and an unattended-Mac boundary. No adapter exists today. | Store credentials in Keychain; allowlist identities/channels; add expiry, deduplication, rate limits, reconnect tests, and source-aware policy. Begin read-only. |
| Context staleness and target confusion | Chrome activation carries tab/window identity, capture time, and fingerprint, requires complete bounded observation plus one unique exact title, and expires after five seconds. The YouTube session carries stable source tab/window/Home identity, feed and item observation IDs, separate presentation/title completeness, and a 90-second expiry. Ordinals resolve only to displayed positions; titles require one normalized exact match in a complete bounded title observation. Protocol-v3 mutation revalidates the chosen video rather than freezing unrelated feed order or tab presentation. A browser/DOM race can still occur after revalidation and before Chrome completes its single update. | Fail on incomplete title evidence for title selection, ambiguity, missing/expired observations, permission removal, source/Home-route change, selected-item drift, close, or API failure. Permit harmless unrelated lazy-load/reorder and tab-title/index churn. Consume a feed session before dispatch; call navigation once; preserve unknown outcomes after dispatched timeout/disconnect; never retry automatically; dogfood navigation, lazy-load, reorder, reload, suspension, and revoke races. |
| Ambiguous conversational reference | A visible feed can contain up to 20 items, so “Open that YouTube video” alone usually does not identify a target. Build 22 scopes terse numbers, “last,” and bare exact titles to the existing visible session; a pronoun selects directly only for a one-item list. A phrase can still contain ordinal and title evidence for different videos. An LLM could produce a plausible choice but cannot supply missing user intent. | Let registered commands win; refuse conflicts and no-session pronouns; clear on expiry, refresh failure, UI clear, or dispatched open. Offer accessible row buttons. Never guess, fuzzily match, or let a model bind an ambiguous referent or authorize navigation. |
| Stable unpacked-extension identity | Build 21 commits a public manifest key so local builds derive one fixed extension ID and the app can register an exact origin without copied setup data. The public key is deliberately reproducible and is not proof of publisher identity. | Keep the private key absent from source, build products, diagnostics, and release tooling. Treat this as development identity only. Use a protected signing/distribution process before Chrome Web Store publication and review any identity change with native-host migration and conflict handling. |
| Focused-field dictation compatibility | Build 19 source recovers focus from only the current frontmost application when the system-wide lookup is unavailable and revalidates its PID. It permits a Codex/ChatGPT caret-at-start replacement only for full suggestion attribution, an empty full text-marker range, or the exact bounded app-owned suggestion observed in dogfooding; all fixed signals and field state are revalidated before mutation. It permits Notion start/middle insertion only for a bounded, single-line, uniformly presented value. A verified whole-value path may reassert only the expected caret and requires delayed stable readback without repeating the value write. Ordinary authored/mixed/marked/unavailable Codex evidence and multiline/rich/object-bearing Notion content fail closed. The exact suggestion fallback could collide with user-authored text equal to that UI copy, so it remains app-scoped, exact, length-bounded, caret-at-start-only, and post-write verified. Terminal remains an explicit fallback; VS Code depends on a writable Accessibility surface and may require screen-reader optimized mode. Live Build 19 cross-app acceptance remains open. | Test empty and nonempty end-append Codex/ChatGPT composers; exact suggestion and authored caret-at-start values; app switches during capture; VS Code with screen-reader mode on/off; Terminal explicit fallback; plain Notion start/middle/end insertion; native AppKit fields; Chrome controls; rich/mixed formatting, links, mentions, lists, attachments; IME/non-Latin text; emoji; and selection replacement. Confirm one verified mutation in supported surfaces, zero mutation on evidence drift, stable post-insertion caret placement, no suggestion contamination or formatting loss, correct fixed preparation/semantic evidence, and no delayed-host duplicates. |
| Dictation polish overreach | Build 11 removes a bounded adjacent restart. Build 14 adds only a timed short-pause “. And” continuation, strong-token spoken slash, and unique vocabulary-equivalent Apple alternative selection. Build 19 reuses only the same uppercase-token slash rule for extracted web-search payloads and adds observed developer corrections only behind exact Apple-alternative corroboration. Even deterministic rewriting can erase intentional wording, while aggressive filler, grammar, or punctuation changes could corrupt code and proper nouns. | Preserve long pauses, likely new-sentence subjects, literal lowercase “slash,” unrelated alternatives, punctuation/newline boundaries, numbers, and acronyms; never polish recovered partials; retain raw versus polished diagnostics but never timing evidence; rate each typed change reason; require zero semantic changes and measured latency before widening the layer. |
| Accidental submission or message send | Focused-field dictation or reply assistance could turn generated text into an externally visible action. | Keep command and dictation modes distinct; dictation never presses Return; separate draft/insert/send capabilities; preview and confirm sends. |
| Sensitive screen/context capture | Chrome tab titles/URLs and YouTube feed titles/channels can reveal private interests even without screenshots or full page bodies. Build 22 continues to exclude incognito, account data, comments, descriptions, likes/subscriptions, arbitrary URLs, screenshots/OCR, persistence, and ordinary logs. Feed values live only in the 90-second in-memory session and visible result. Up to 60 bounded ID/title candidates may exist only inside one extractor invocation to prove selection uniqueness; no additional candidate strings cross the native protocol. Content-free readiness reads no tab/page state. The explicit dogfood trace may retain a title only when the user authors it in a command; it never appends returned feed data. | Keep acquisition on demand and bounded; verify logs/exports/UserDefaults contain no browser-returned values; preserve the developer trace's user-authored-command-only exception; make permission state/removal and UI clearing obvious; require a new decision before another DOM schema, origin, screenshot, cloud use, or persistence. |
| Credential disclosure | Future remote/model providers will require secrets; a public repository and diagnostics increase exposure risk. | Keep credentials in Keychain; never source/plist/defaults/logs; scan every diff and reachable history before publishing; use GitHub secret protection as defense in depth. |
| Toolchain reproducibility | Xcode 26.6 is selected; SwiftPM tests and Debug/Release app builds pass. | Keep package pins and shared scheme; add a clean-machine or CI build before wider distribution. |
| Local signing and distribution | Release is hardened and ad-hoc signed; strict `codesign` verification passes, but it has no Developer ID or notarization. Ad-hoc identity is not a basis for distribution or durable TCC assumptions. | Treat this build as local-only. Choose the sandbox/entitlement posture deliberately, use Developer ID signing, verify permission persistence, and notarize before sharing it with other Macs. |
| Shortcut conflicts | No default chord can be assumed safe. | User-recorded standard modified shortcut and conflict UI; no Fn/modifier-only interception. |
| Menu-bar lifecycle | The installed app registers as a UI element, creates its status-item scene, remains alive, and passed the initial manual shortcut/command check. | Exercise removal, relaunch, sleep/wake, and onboarding before relying on persistent background use. |
